HWPX 변환 최적화 루프
모든 포맷을 깔때기로 모으고, 검증으로 닫는다

12개 입력 포맷을 단일 중간표현(IR)으로 정규화한 뒤 생성·검증·복구 루프로 닫는 에이전트 아키텍처. 직접 변환기 12개가 아니라, 어댑터 N개 + 생성기 1개 + 검증 루프 1개 구조.

🎯 설계 원칙 — 함정 먼저
"되는 것처럼 보이지만 한글에서 안 열림"을 막는 세 가지 전제.
⚠️
함정 1 · 포맷마다 변환기를 따로 만들면 유지보수 지옥
12개 경로를 개별 구현하면 코드 12벌·버그 12배임. 정답은 모든 입력을 중간표현(IR = Markdown 또는 구조화 JSON)으로 정규화 → HWPX 생성기 1개로 통과시키는 깔때기 구조. 신규 포맷은 "→IR" 어댑터 1개만 추가하면 끝.
⚠️
함정 2 · 검증 루프 없는 "변환"은 절반의 완성
HWPX는 mimetype을 ZIP_STORED로 ZIP 첫 항목에 배치하고 META-INF·Preview 구조를 지켜야 한글에서 열림. 생성만 하고 검증을 안 하면 "변환 성공 → 한글 실행 시 오류"가 반복됨. generate → validate → 시각 확인 → repair가 루프의 본체.
⚠️
함정 3 · PDF·PPTX·EPUB는 무손실이 아님
레이아웃·표 병합·이미지 앵커 손실이 구조적으로 불가피함. 기본 서비스(MD·HTML·DOCX·TXT·CSV)와 동일 품질을 기대하면 실패함. 확장 서비스는 "서식 복제"가 아니라 "내용 추출 + 재조판"으로 기대치를 분리해 명시할 것.
✅
결론 아키텍처
어댑터(N) → IR(1) → 생성기(1) → 검증 루프(1). 입력이 늘어도 생성·검증부는 그대로. 이것이 바이브 코딩에서 "한 번 만들고 계속 확장"이 가능한 유일한 형태.
