1. 결론 먼저 보기

HWPX 자동화는 하나의 정답 저장소로 끝나지 않습니다. 목적별로 “변환기”, “파서”, “문서 생성기”, “MCP 서버”, “업무 자동화 노드”를 조합하는 방식이 가장 빠릅니다.

가장 빠른 실전 조합

md/docx/html → hwpx 웹페이지를 확장하려면 pypandoc-hwpx, md2hwpx, hwpxjs를 우선 비교하십시오. 스타일 템플릿·표 병합·이미지 임베딩이 핵심 검증 항목입니다.

바이브 코딩 최적 조합

문서 읽기·수정·검증을 에이전트에게 맡기려면 python-hwpx + hwpx-mcp-server가 가장 직접적입니다. TypeScript 기반 서비스라면 hwpx-ts 또는 kordoc 쪽이 유리합니다.

2. 목적별 추천 스택

1문서 변환 웹서비스MD/DOCX/HTML 입력 → HWPX 출력.

2공공문서 자동 작성보고서 문체 정리, 결재 양식 채우기, 템플릿 보존.

아래 프롬프트는 Cursor, VS Code/Codex, Claude Code, Gemini CLI 등에서 저장소를 붙여놓고 바로 지시할 때 쓰기 좋습니다.

A. md/docx/html → HWPX 통합 변환기 만들기

역할: HWPX 변환 웹서비스의 시니어 엔지니어로 작업한다.
목표: Markdown, DOCX, HTML 입력을 HWPX로 변환하는 통합 API를 만든다.
참조 저장소: msjang/pypandoc-hwpx, JunDamin/md2hwpx, ssabro/hwpxjs.
요구사항:
1. 입력 파일 형식을 자동 감지한다.
2. Markdown은 md2hwpx 우선, DOCX/HTML은 pypandoc-hwpx 우선으로 처리한다.
3. 표 병합(rowspan/colspan), 이미지 임베딩, 제목 계층, 각주를 테스트 케이스로 만든다.
4. 변환 실패 시 원인, 중간 산출물(JSON AST 또는 HTML), 사용자 조치 메시지를 반환한다.
5. FastAPI 기준 /convert, /validate, /preview 엔드포인트를 구현한다.
6. 샘플 파일 5개로 회귀 테스트를 만든다.

B. HWPX 템플릿 자동 채우기

역할: 공공기관 HWPX 양식 자동화 엔지니어로 작업한다.
목표: template.hwpx의 {{필드명}} 플레이스홀더를 JSON 값으로 채우고, 원본 서식을 최대한 유지한다.
참조 저장소: airmang/python-hwpx, neolord0/hwpxlib, chrisryugj/kordoc, Kminer2053/public-doc-to-hwpx.
구현 지시:
1. HWPX는 ZIP 패키지로 안전하게 풀고 XML 경로를 제한한다.
2. 모든 XML에서 플레이스홀더 목록을 추출하는 inspect_template()를 만든다.
3. fill_template(template_path, data, output_path)를 만든다.
4. 적용 성공/실패 필드를 applied, missing, unused로 리포트한다.
5. 한글 프로그램 없이 동작해야 한다.
6. 원본 파일은 절대 덮어쓰지 말고 복사본을 만든다.

C. MCP로 HWPX 도구화하기

역할: MCP 서버 설계자로 작업한다.
목표: AI 클라이언트가 HWPX를 읽고, 검색하고, 수정하고, 검증할 수 있는 MCP 도구를 만든다.
참조 저장소: airmang/hwpx-mcp-server, aliencube/mcp-pypandoc-hwpx, ubermensch1218/hwpx-ts.
도구 목록:
- hwpx_read(file): 텍스트와 표 요약 반환
- hwpx_export_markdown(file): Markdown 반환
- hwpx_find_placeholders(file): {{필드}} 목록 반환
- hwpx_fill_template(file, data): 복사본 생성 후 값 치환
- hwpx_validate(file): 패키지 구조와 XML 파싱 오류 반환
주의사항:
- 파일 경로는 허용된 workspace 안으로 제한한다.
- 원본은 변경하지 않는다.
- 모든 수정 작업은 diff 또는 적용 리포트를 반환한다.

6. AI Agent로 요약·보고문 문체 변환
7. HWPX 템플릿 채우기
8. 완성 파일 저장 및 메일 회신

5. 현재 md → hwpx 웹페이지 확장 판단

같이 묶는 것이 좋은 경우

사용자 입장에서는 “문서를 HWPX로 바꾸는 서비스” 하나로 인식합니다. MD, DOCX, HTML을 같은 UI에 두고 입력 형식만 자동 감지하면 사용성이 좋습니다.

내부 구조는 분리

엔진은 md2hwpx, pypandoc-hwpx, hwpxjs처럼 어댑터 패턴으로 분리하십시오. UI는 하나, 변환 엔진은 플러그인형이 안전합니다.

검증 포인트

표 병합, 이미지, 각주, 제목 계층, 참조 템플릿 스타일, 한컴 열기 호환성, 변환 실패 리포트까지 자동 테스트해야 합니다.

권장 아키텍처
단일 웹페이지 + 내부 변환 어댑터 분리. 즉, 사용자는 “HWPX 변환기” 하나로 쓰고, 개발자는 md/docx/html 엔진을 분리해 유지보수하는 구조가 가장 좋습니다.

7. 구현 시 주의사항

보안

HWPX는 ZIP 패키지이므로 ZIP Slip 방어, 압축 폭탄 제한, XML 외부 엔티티 차단, 허용 경로 제한이 필요합니다. MCP 서버나 웹 업로드 API에서는 원본 덮어쓰기를 금지해야 합니다.

품질

“열린다”와 “서식이 유지된다”는 다릅니다. 표 병합, 이미지, 글꼴, 머리말·꼬리말, 각주, 페이지 나눔, 참조 템플릿 스타일을 테스트해야 합니다.

라이선스

MIT, Apache-2.0 계열이 많지만 저장소마다 다릅니다. 상용·기관 배포 전 LICENSE 파일을 확인하고, 한컴 공개 문서 참고 문구가 필요한 구현체는 고지문을 보존하십시오.

운영

웹서비스는 변환 엔진별 timeout, 변환 로그, 중간 산출물, 실패 파일 보관 정책을 두어야 합니다. 바이브 코딩으로 만든 기능일수록 회귀 테스트가 품질의 핵심입니다.
