# To HWPX 디자인 지침

이 문서는 To HWPX의 화면을 고칠 때 유지할 공통 기준이다. 목표는 화려한 랜딩페이지가 아니라, 사용자가 변환 전 상태를 빠르게 이해하고 믿고 실행할 수 있는 문서 작업 도구의 품질이다.

## 디자인 원칙

- 첫 화면의 주 행동은 파일 선택이다. 보조 안내, 설치, 개인정보, 업데이트 버튼은 파일 선택보다 먼저 시선을 빼앗지 않는다.
- 변환 화면은 `입력 → 변환 전 문서 모양 → 세부 설정 → 실행 → 결과` 순서로 읽혀야 한다.
- 자주 바꾸는 값은 숨기지 않는다. 글꼴, 글꼴 크기, 용지 크기, 용지 방향은 변환 전 문서 모양 영역에 항상 노출한다.
- 드문 값은 접는다. 상단 제목 블록, 페이지 여백, 자동 다운로드, 개발자용 IR 미리보기는 세부 설정 안에 둔다.
- 설명은 짧게 쓰고, 사용자가 지금 어떤 값으로 변환하는지 보여준다.

## 시각 톤

- 공공 문서, 보고서, 한글 문서 편집 맥락에 맞는 조용한 고급감을 우선한다.
- 큰 장식, 과한 그라데이션, 마케팅식 히어로, 중첩 카드, 장식용 배경 오브젝트는 쓰지 않는다.
- 패널 안에서는 두꺼운 카드보다 얇은 구분선, 간결한 헤더, 안정적인 입력 밀도로 위계를 만든다.
- 색상은 상태와 행동을 구분하는 데 쓴다. 한 화면을 한 가지 색 계열로만 밀지 않는다.
- 버튼 문구는 결과와 연결한다. 예: `이 설정으로 HWPX 만들기`.

## 변환기 패널 구조

권장 구조:

```text
파일 선택

문서 제목
포맷 보존/확인 안내

변환 전 문서 모양
[한글 폰트] [글꼴 크기] [용지 크기] [용지 방향]

세부 설정
상단 제목 블록 / 페이지 여백 / 자동 다운로드

[이 설정으로 HWPX 만들기]
```

금지 구조:

```text
파일 선택
문서 제목
접힌 고급 설정 안에 주요 옵션 전체
변환 시작
```

## 컴포넌트 기준

- 드롭존은 가장 강한 입력 affordance를 가진다. 선택 후에는 파일 상태와 형식이 바로 보여야 한다.
- 포맷 힌트는 보존/확인 정보를 알려주되, 설정 영역 안내와 경쟁하지 않는다.
- 변환 전 문서 모양 영역은 접지 않는다. 현재값 요약과 실제 컨트롤을 함께 보여준다.
- 세부 설정 summary는 보조 기능임을 드러낸다. `더 보기` 수준의 낮은 위계가 적절하다.
- 모바일에서는 4열 입력을 1열 또는 2열로 줄이고, 텍스트가 버튼/입력 안에서 넘치지 않게 한다.

## 문구 기준

- `고급 설정`이라는 표현은 초심자에게 숨은 기능처럼 보이므로 피한다.
- 기본 표현은 `변환 전 문서 모양`, 보조 접힘 영역은 `세부 설정`을 쓴다.
- 기능 설명보다 현재 행동을 말한다.
  - 좋음: `이 값으로 HWPX가 만들어집니다.`
  - 피함: `글꼴, 글꼴 크기, 용지, 여백, 자동 다운로드를 설정합니다.`
- HWPX는 출력 형식으로 설명한다. 입력 가능 포맷처럼 쓰지 않는다.

## 롤백 가능한 작업 방식

- 옵션의 `id`, `name`, `data-*`, localStorage 키, 변환 state 이름은 바꾸지 않는다.
- UI 재배치와 변환 로직 변경은 같은 커밋에 섞지 않는다.
- 주요 설정 노출은 DOM 이동과 CSS 변경 중심으로 처리한다.
- 새 디자인이 마음에 들지 않으면 해당 UI 커밋만 `git revert`할 수 있어야 한다.
- 버전 범프와 changelog는 최종 확인 후 마지막 커밋에 반영한다.

## QA 체크리스트

- 첫 사용자가 5초 안에 글꼴/용지/방향 설정 위치를 찾을 수 있는가.
- 파일 선택 후 현재 변환값이 화면에 보이는가.
- 모바일 390px 폭에서 설정 컨트롤과 변환 버튼 텍스트가 넘치지 않는가.
- 다크 테마에서 구분선, 배지, 버튼 대비가 충분한가.
- `node qa/gate.js qa/fixtures/md_hwpx_test.md`와 `npm.cmd run test:golden`이 통과하는가.
